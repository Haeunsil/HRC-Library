# CodeBook 표시용 코드 체계 설계안 (raw 유지 · 웹 전용)

행정동·법정동 **원본 코드(raw)**는 공공·내부 식별자로 그대로 두고, **웹·리포트·연계 시스템**에서만 쓰는 **표시용(display) 코드**를 별도 계층으로 두는 방안을 정리한다.  
구현 여부와 무관하게, 요구 조건(시도 1–17 고정, 시군구·읍면동 재구성, 행정/법정 일관성)을 만족시키기 위한 **권장 설계**다.

---

## 0. 요구 조건 정리

|---|------|

## 1. raw 유지 + 웹 전용 코드 체계를 **별도로** 설계하는 방법

### 1.1 이중 계층 모델

- **Canonical(원본) 계층**  
  - `raw_code` (10자리 행정기관·법정동 등), `시도명`·`시군구명`·`읍면동명`, `기준연월` 등 **수집 그대로**.  
  - 역할: **불변 식별자**, 재수집·감사·공공 API 대조, 기존 배치·스냅샷과의 호환.

- **Presentation(표시) 계층**  
  - `disp_sido` (1–17), `disp_sigungu`, `disp_eup` 등 **사내 규칙으로 부여한 코드**.  
  - 역할: **UI·엑셀·URL 파라미터·타 시스템 연계**에만 사용.

### 1.2 구현 위치 (권장)

- **서버**: 매핑 테이블 + API가 `raw`와 `display`를 **한 번에** 내려주거나, 클라이언트가 조회 전용 API로 `display`만 받는다.  
  - 이유: 브라우저만 규칙을 들고 있으면 **엑셀·다른 클라이언트**와 불일치하기 쉽다.
- **클라이언트**: 표시·정렬·필터는 `display` 우선, 상세/다운로드 옵션으로 **raw 노출** 유지.

### 1.3 절대 하지 말 것

- raw 컬럼을 표시용 값으로 **덮어쓰기**.  
- 표시용 규칙만 바꿨다며 **스냅샷 적재 시 raw를 잘라 쓰기** (식별자 손실).

## 2. 시도를 1–17로 고정했을 때, 시군구·읍면동 **재매핑** 규칙 제안

행정표준 **10자리 앞 2자리**와 1–17 체계는 **숫자가 다르다**(예: 서울 `11` vs 표시 `1`). 따라서 **시군구·읍면동을 raw 자리만 잘라 붙이면** 의미 있는 “새 체계”가 되지 않는다.  
권장은 **「시도 = 명칭·고정표 매핑」+「시군구·읍면동 = 계층 내 서로게이트」** 조합이다.

### 2.1 시도 (`disp_sido`)

- **1차**: `시도명` 정규화(공백·특별자치 명칭 통일) 후 **위 고정표로 매핑**.  
- **보조**: 명칭 누락·오탈 시에만 `raw` 앞 2자리 → **별도 Kostat↔1–17 대조 테이블**로 역매핑(운영 부담이 있으므로 1차는 명칭 권장).

### 2.2 시군구 (`disp_sigungu`) — ruleset **v3** (현행)

- **의미**: **4자리 문자열 고정**. `disp_sido`를 정수 `d`로 두고, 시군구 라벨 순번 `s=1..99`에 대해 앞 2자리 `p`와 뒤 2자리 `ss`를 붙인다(`p`+`ss` = `f"{p}{s:02d}"`).  
  - `d`가 1–9: `p = 10×d` → 서울 `d=1`이면 `1001`–`1099`, `d=9`이면 `9001`–`9099`.  
  - `d = 10`: `p = 19` → `1901`–`1999`(시도 10과 시도 1의 `10xx` 대역 충돌 방지).  
  - `d`가 11–17: `p = d`(두 자리) → 예 `d=14`이면 `1401`–`1499`.  
- **키**: `(disp_sido, 시군구명)` — 세종 등은 `세종시` 통일 후 동일.  
- **한도**: 시도당 시군구 **최대 99**개(`s` 01–99). 초과 시 `ValueError`.  
- **중복 방지**: `(disp_sido, disp_sigungu)` 쌍으로 식별.

### 2.3 읍면동 (`disp_eup`) — ruleset **v3** (현행)

- **의미**: **6자리 문자열 고정**. `n = int(disp_sigungu) × 100 + k`(동일 시군구 내 읍면동 라벨 순번 `k=1..99`). `disp_eup = f"{n:06d}"`.  
  - 예: `disp_sigungu="1001"`이면 `100101`–`100199`, `"1401"`이면 `140101`–`140199`.  
- **키**: `(disp_sido, 시군구 키, 읍면동명)`.  
- **한도**: 시군구당 읍면동 **최대 99**개. 초과 시 `ValueError`.  
- **정렬·부여 순서**: §2.2와 동일하게 라벨 정렬 후 부여, 응답 `items`는 시도 코드 → 시군구 라벨 → 읍면동 라벨 → raw 순.

**한글 정렬(서버)**: NFKC + 튜플 정렬. 클라이언트는 `localeCompare('ko', { numeric: true })` — `client/src/utils/codebookDisplayCodesV2.js` 의 `enrichDisplayCodesV3FromSido`.

### 2.4 표시용 “복합 키” (선택)

- `disp_sigungu`(4) + `disp_eup`(6) 조합으로 계층을 읽을 수 있다(`disp_eup` 앞 4자리가 시군구 코드와 동일).

## 3. 행정동·법정동을 **함께** 관리할 때 충돌 없는 구조

### 3.1 원칙

- **같은 raw 문자열**이 행정과 법정에 **동시에 존재**할 수 있으므로, 매핑 PK는 **`(kind, raw_code)`** 를 권장한다.  
  - `kind ∈ { 'admin', 'legal' }`

### 3.2 테이블 분리 vs 통합

| 방식 | 장점 | 단점 |
|------|------|------|
| **테이블 분리** `display_map_admin`, `display_map_legal` | 스키마·인덱스 단순, 배치 독립 | 공통 뷰·리포트에서 UNION 필요 |
| **단일 테이블** `display_map(kind, raw_code, …)` | 조회 단순 | 인덱스·배치 잠금 고려 |

**권장**: **단일 테이블** + `(kind, raw_code)` 유니크. 행정·법정 **같은 표시 규칙 함수**를 쓰되, 입력 row는 kind별로 분리.

### 3.3 “같은 지역” 정합성

- 행정·법정은 **행정구역 단위가 1:1이 아닐 수 있음**.  
- 정책 선택:  
  - **A)** 표시 코드를 **완전 독립** 부여(추천: 충돌·법적 혼선 최소).  
  - **B)** `(시도명, 시군구명, 읍면동명)` 정규화 키로 **공통 서브테이블**을 두고, kind별 raw는 그 공통 ID를 FK로 참조(설계 복잡도↑).

초기에는 **A**로 가고, 필요 시 **B**로 진화하는 것이 운영 리스크가 낮다.

## 4. raw ↔ 웹 표시용 **매핑 테이블** 관리

### 4.1 권장 스키마 (논리)

| 컬럼 | 설명 |
|------|------|
| `kind` | `admin` \| `legal` |
| `raw_code` | 원본 10자리(또는 정규화된 문자열) |
| `disp_sido` | 1–17 |
| `disp_sigungu` | ruleset v3: **4자리** 고정(§2.2) |
| `disp_eup` | ruleset v3: **6자리** 고정(§2.3) |
| `label_sido`, `label_sigungu`, `label_eup` | 매핑 시점 스냅샷 명칭(디버깅·감사) |
| `source_snapshot_id` / `refreshed_at` | 어느 스냅샷·배치에서 생성됐는지 |
| `mapping_version` | 규칙 변경 시 증가 |

**유니크**: `(kind, raw_code)`  
**조회**: UI는 `kind` + 목록 API에서 `raw + display` 동시 반환.

### 4.2 생성 파이프라인

1. odcloud/스냅샷에서 **raw 행 적재**(기존과 동일).  
2. 배치(또는 트리거)가 `시도명` → `disp_sido` 매핑 후, 시군구·읍면동 **일련 부여**로 `display_map` upsert.  
3. API: CodeBook 조회 시 **JOIN** 또는 **서버에서 merge**하여 한 레코드로 반환.

### 4.3 버전

- `mapping_version` 또는 `ruleset_id`를 두고, **규칙 변경 시** 재생성 배치를 돌린다.  
- 이전 버전 테이블을 보존하면 **롤백·비교**가 가능하다.

## 5. 코드북 변경 시 **기존 데이터 호환성**

| 전략 | 내용 |
|------|------|
| **raw 고정** | 외부·내부 레거시는 계속 `raw`로 조인 가능. |
| **이중 표기 기간** | UI에 `표시용` + `(원본)` 병기, 엑셀에 컬럼 분리 `raw_*` / `disp_*`. |
| **API 버전** | `?display=v2` 또는 헤더로 응답 스키마 분기. |
| **매핑 재생성** | 명칭 개칭만으로 raw가 같으면 **표시 코드 유지**를 기본 정책으로 두고, **불가피할 때만** 재부여 + 변경 로그. |

## 6. 예외 케이스 처리 (세종·누락 명칭·통폐합)

### 6.1 세종특별자치시

- 많은 데이터에서 **시군구명이 비어 있거나 시도와 동일**한 식으로 온다.  
- **서버(CodeBook)**: 시도가 세종으로 판별되면(`apply_sejong_unified_sigungu`) **`시군구명`·`sigungu_name`을 일괄 `세종시`로 통일**한다(odcloud 수집·스냅샷 읽기·표시 코드 enrich·`/api/address/codebook` 공통).  
- 처리:  
  - `disp_sido = 17` 고정 후, 시군구 키는 위 통일명 기준으로 **단일 구역**(v3에서는 `disp_sigungu = 17001` 등 구간 `17001`~`17999`의 첫 번호).  
  - 읍면동은 그 아래 3자리 일련.

### 6.2 명칭 누락·불일치

- `시도명` 비어 있으면: raw 앞 2자리 → **보조 매핑 테이블** → 실패 시 `disp_sido = NULL` + **검증 리포트**에 올림(UI에는 “미매핑”).

### 6.3 통·폐합

- raw가 바뀌면 **새 행**이므로 `(kind, raw_code)`로 **새 display 행** 생성.  
- **옛 raw**는 매핑 행을 **deprecated** 플래그로 남기고, 필요 시 “승계 raw” 링크 컬럼 추가.

## 7. 권장 코드 체계 요약 · 샘플 매핑

### 7.1 권장 체계 (한 줄)

- **raw**: 공식·수집 식별자, **절대 표시 규칙에 묶지 않음**.  
- **disp_sido**: **시도명 → 1–17** 고정표.  
- **disp_sigungu / disp_eup**: **시도(및 시군구) 범위 안에서 3자리 서로게이트**, 부여 규칙은 **버전 고정 문서**로 관리.  
- **저장**: `(kind, raw_code)` 유니크 매핑 테이블 + **mapping_version**.

### 7.2 샘플 (행정기관 raw 예: `1111051500`, 서울 종로 청운효자동)

| 필드 | 값 |
|------|-----|
| kind | `admin` |
| raw_code | `1111051500` |
| disp_sido | `1` |
| disp_sigungu | `015` *(예: 서울 내 종로구에 부여된 일련)* |
| disp_eup | `042` *(예: 해당 시군구 내 일련)* |
| 복합 표시(선택) | `01-015-042` |

*(015·042는 예시이며, 실제 운영에서는 스냅샷 전체 정렬 후 부여한 값이 된다.)*

### 7.3 샘플 (세종, 시군구명 공란 → 서버에서 `세종시` 통일, v3)

| raw_code (예시) | 시도명 | 시군구명(저장/표시) | disp_sido | disp_sigungu | disp_eup |
|-----------------|--------|---------------------|-----------|--------------|----------|
| … | 세종특별자치시 | 세종시 | 17 | 1701 | (첫 읍면동이면 `170101` 형태) |

## 8. 운영 시 주의사항

1. **활용신청·키**: odcloud 15097972 / 15099158 등 **원본 수집**과 표시용 매핑은 별개이나, **입력 데이터 변경** 시 매핑 배치를 재실행할지 정책을 문서화한다.  
2. **일관성**: 시군구·읍면동 부여 알고리즘은 **코드·SQL·문서** 세 곳에 중복 서술하지 말고, **한 곳(예: 서버 모듈 + 본 문서 링크)** 을 기준으로 한다.  
3. **검증 배치**: 매핑 후 `disp_sido IS NULL` 또는 **동일 복합키 중복** 검출 리포트를 자동화한다.  
4. **엑셀 다운로드**: `raw_*` 컬럼을 기본 포함해 **감사 추적**을 쉽게 한다.  
5. **법·행정 해석**: 표시용 코드는 **내부 업무 규칙**이며, 통계 제출 등에는 **공식 코드**를 따로 쓰는지 조직 규정에 명시한다.

### 9.1 샘플 입·출력 (v3, 서울 `disp_sido=1` 가정)

**입력** — 시도 코드 `1`, 시군구·읍면동 라벨만 예시.

| 강동구 | … |
| 강북구 | … |

**시군구 코드** — 라벨 순: 강남→강동→강북… → `1001`, `1002`, `1003`, …

**읍면동 코드** — 시도 `1`, 시군구 `1001`(강남)일 때 읍면동 라벨 순:

| 읍면동명 | disp_sigungu | disp_eup |
|----------|----------------|----------|
| 개포1동 | 1001 | 1100101 |
| 개포2동 | 1001 | 1100102 |
| 대치1동 | 1001 | 1100103 |

---

**한계(의도)**: 시군구·읍면동 표시 코드는 **해당 응답에 포함된 행 집합** 기준으로 정렬·일련 부여한다. 스냅샷 행이 늘거나 줄면 **같은 명칭이라도 일련이 바뀔 수 있다**. 영구 고정 번호가 필요하면 본 문서 §4의 **매핑 테이블 영속화** 단계로 진화시킨다.
